# CloudBuddy — план развития (для агента / поэтапная разработка)

Этот документ задаёт порядок внедрения фич и изменений в архитектуру. Агент или разработчик может следовать этапам по очереди, чтобы не ломать уже сделанное и минимизировать конфликты.

---

## Этап 0: Текущее состояние (сделано)

- [x] Android-проект на Kotlin + Jetpack Compose.
- [x] Один экран — мини-игра Flappy Bird (Canvas, тап = прыжок, препятствия-облака, счёт, Game Over).
- [x] Тема CloudBuddy (Material 3).

---

## Этап 1: Навигация и разделение экранов

**Цель:** отделить «мир питомца» от «игры», чтобы потом вставить чат и логику питомца.

- [ ] Добавить зависимость **Compose Navigation**.
- [ ] Ввести два маршрута (например): `home` (главный экран с питомцем), `game` (мини-игра).
- [ ] Вынести текущую игру в отдельный экран/граф (например, `GameRoute`), вызываемый по `game`.
- [ ] Создать **главный экран (Home)** — заглушка: облачко (картинка или простой Composable), кнопки «Чат» и «Играть» / «Выгулять».
- [ ] MainActivity: `NavHost` с стартовым маршрутом `home`; с `game` открывается экран игры.
- [ ] По завершении игры (Game Over или «Выход») — возврат на `home`.

**Результат:** при запуске пользователь видит главный экран; игра доступна по кнопке и возвращает на главный.

---

## Этап 2: Состояние питомца (настроение, скука)

**Цель:** облачко имеет «состояние», от которого потом будет зависеть приглашение играть и текст в чате.

- [ ] Ввести простую модель состояния питомца (например, data class): настроение (enum или Int), уровень скуки, последнее время кормления/игры (timestamp).
- [ ] Хранить состояние в **ViewModel** главного экрана (или общем ViewModel/StateHolder), пока без персистенции.
- [ ] На главном экране отображать индикатор настроения/скуки (иконки, текст или полоска).
- [ ] Реализовать простой таймер/фоновое правило: со временем «скука» растёт (например, раз в N минут), при входе в игру — уменьшается.
- [ ] Опционально: при высоком уровне скуки показывать подсказку «Облачку скучно — поиграй со мной!» на главном экране.

**Результат:** питомец имеет настроение и скуку; скука растёт со временем и снижается после игры.

---

## Этап 3: Локальное сохранение состояния

**Цель:** после перезапуска приложения состояние питомца и (опционально) последний счёт игры не теряются.

- [ ] Подключить **DataStore** (Preferences) или Room (если решите хранить историю чата локально позже).
- [ ] Сохранять: уровень скуки, настроение, время последнего взаимодействия; при старте приложения — загружать и передавать в ViewModel.
- [ ] При изменении состояния питомца (игра закончена, прошло время) — сохранять в DataStore.
- [ ] Опционально: сохранять лучший счёт игры и показывать его на экране игры или дома.

**Результат:** состояние питомца переживает перезапуск приложения.

---

## Этап 4: Интеграция OpenAI (чат с облачком)

**Цель:** пользователь может общаться с облачком через текстовый чат, ответы генерирует OpenAI.

- [ ] Добавить зависимость для HTTP (Retrofit + OkHttp или Ktor).
- [ ] Создать **API-клиент** для OpenAI Chat API (или совместимого endpoint): отправка сообщений, получение ответа; ключ API хранить в BuildConfig / local.properties / env, не коммитить.
- [ ] Ввести экран/экранчик **чата**: поле ввода, список сообщений (роль + текст). Состояние чата — в ViewModel (список сообщений).
- [ ] При отправке сообщения: добавить его в список, вызвать API, добавить ответ ассистента в список; обработать ошибки и загрузку.
- [ ] Системный промпт задать так, чтобы модель «играла» облачко: дружелюбное, иногда упоминает скуку и игру («давай поиграем», «выгуляй меня»).
- [ ] Опционально: передавать в API текущее состояние питомца (настроение, скука), чтобы ответы зависели от контекста.

**Результат:** пользователь может вести диалог с облачком через OpenAI; ответы контекстные (питомец, скука, игра).

---

## Этап 5: Связь чата, питомца и игры

**Цель:** единый цикл: питомец скучает → пользователь видит это в чате и на главном экране → идёт играть → скука падает, настроение улучшается.

- [ ] При открытии чата передавать в промпт (или в сообщения) текущие значения скуки и настроения.
- [ ] После игры (завершение сессии, счёт) обновлять состояние питомца: уменьшать скуку, возможно улучшать настроение в зависимости от счёта.
- [ ] На главном экране: при высокой скуке подсвечивать кнопку «Играть» или показывать реплику облачка («Поиграй со мной!»).
- [ ] В чате: если скука высокая, модель может чаще предлагать поиграть; после игры — благодарить и комментировать результат.

**Результат:** чат, настроение и игра образуют одну связанную механику.

---

## Этап 6: Улучшение игры и полировка

**Цель:** игра стабильна, понятна и визуально в духе «облачко».

- [ ] Снизить `@RequiresApi` до минимально нужного (или убрать, если не используется API 35) для совместимости.
- [ ] Добавить экран паузы, кнопку «Выход» в игру (возврат на главный экран с сохранением результата для состояния питомца).
- [ ] Улучшить графику: заменить примитивы на спрайты/картинки (облачко-персонаж, облака-препятствия), фон неба.
- [ ] Звуки: прыжок, проход препятствия, Game Over (опционально).
- [ ] Анимации: плавное появление/исчезновение экранов, простые анимации облачка на главном экране.

**Результат:** игра выглядит и ощущается как часть «облачного» тамагочи.

---

## Этап 7: Дополнительные фичи (по желанию)

- [ ] **Уведомления:** напоминание «Облачку скучно» через несколько часов неактивности (WorkManager + канал уведомлений).
- [ ] **Статистика:** сколько раз играли, лучший счёт, «дни подряд» захода.
- [ ] **Настройки:** включение/выключение звука, выбор модели OpenAI, тема (светлая/тёмная/системная).
- [ ] **Офлайн-режим:** без API показывать заготовленные фразы облачка и всё равно позволять играть и менять скуку.

---

## Рекомендации для агента

1. Выполнять этапы **по порядку**: 1 → 2 → 3 → 4 → 5 → 6 → 7.
2. В каждом этапе делать небольшие коммиты (навигация, ViewModel, API, UI).
3. После Этапа 1 не ломать существующую игру: она должна оставаться в отдельном маршруте.
4. Ключи и секреты (OpenAI API key) не коммитить; использовать `local.properties` или переменные окружения и документировать в README.
5. При добавлении новых экранов и слоёв обновлять **docs/ARCHITECTURE.md** (новые пакеты, экраны, зависимости).

После выполнения этапов приложение будет полноценным «тамагочи с ИИ»: облачко-друг, чат с OpenAI, скука и «выгул» через Flappy-подобную игру.
